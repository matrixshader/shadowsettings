interactive Adwaita row widget, with reset-to-default and changed-settings highlighting on every row.

Purpose: This is the core engine that transforms raw SettingDef metadata into interactive UI controls.
Without it, the app shows placeholder ActionRows with text labels only. With it, every detected
hidden setting becomes a fully interactive widget.

Output: `src/core/widget-factory.vala` containing WidgetFactory class with `create_row()` entry point.

<task type="auto">
  <name>Task 1: Create WidgetFactory with all widget creators and reset/changed highlighting</name>
  <files>src/core/widget-factory.vala</files>
  <action>
Create `src/core/widget-factory.vala` containing a `WidgetFactory` class in the `ShadowSettings` namespace.

**Class structure:**
- Static class with a single public entry point: `public static Gtk.Widget? create_row (SettingDef def, SchemaScanner scanner)`
- Private static `HashTable<string, GLib.Settings>` for GSettings object cache (Pitfall 1 from research)
- Private static helper: `get_cached_settings(schema_id)` that checks cache, falls back to `SafeSettings.try_get()`, stores result

**create_row dispatch:**
Switch on `def.widget_hint`:
- `SWITCH` -> `create_switch_row(def, settings)`
- `COMBO` -> `create_combo_row(def, settings)`
- `SPIN_INT` -> `create_spin_int_row(def, settings)`
- `SPIN_DOUBLE` -> `create_spin_double_row(def, settings)`
- `FONT` -> `create_font_row(def, settings)`
- `ENTRY` -> `create_entry_row(def, settings)`
- `AUTO` -> `create_auto_row(def, settings, scanner)`
- `CUSTOM` -> return null (handled separately, e.g. PowerPanel)

If `get_cached_settings()` returns null, return null (schema not on this system).

**Widget creators (all private static):**

1. `create_switch_row`: Create `Adw.SwitchRow`, set title/subtitle from def, use `settings.bind(def.key, row, "active", SettingsBindFlags.DEFAULT)` for two-way sync. Call `add_reset_button()` and `track_modified_state()`.

2. `create_combo_row`: Create `Adw.ComboRow`, build `Gtk.StringList` from `def.combo_labels`, set model, find initial selection by matching `settings.get_string(def.key)` against `def.combo_values`. Wire `notify["selected"]` to `settings.set_string()`. Handle case where current value doesn't match any combo_values (leave at 0 -- Pitfall 4). Call reset/modified helpers.

3. `create_spin_int_row`: Create `Adw.SpinRow.with_range(def.spin_min, def.spin_max, def.spin_step)`, set `digits = (uint)def.spin_digits`, set initial `value = settings.get_int(def.key)`. If `def.display_factor != 0.0`, multiply displayed value by display_factor and divide on write-back. Wire `notify["value"]` to `settings.set_int()` (cast from double). Call reset/modified helpers. **IMPORTANT:** Use separate methods for int vs double (Pitfall 3) -- int uses `get_int()`/`set_int()`, double uses `get_double()`/`set_double()`.

4. `create_spin_double_row`: Same as spin_int but uses `settings.get_double()`/`settings.set_double()`.

5. `create_font_row`: Create `Adw.ActionRow` (NOT SwitchRow/ComboRow -- ActionRow is the base), set title/subtitle. Create `Gtk.FontDialogButton(new Gtk.FontDialog())`, set `valign = CENTER`. Initialize from `Pango.FontDescription.from_string(settings.get_string(def.key))`. Wire `notify["font-desc"]` to `settings.set_string(font_button.font_desc.to_string())`. Use `row.add_suffix(font_button)` THEN call reset/modified (reset button added after font button). Call reset/modified helpers.

6. `create_entry_row`: Create `Adw.EntryRow`, set title (subtitle goes as tooltip since EntryRow has no subtitle property -- use `row.tooltip_text = def.subtitle` if non-null). Set `show_apply_button = true`. Initialize with `row.set_text(settings.get_string(def.key))`. Wire `apply` signal to `settings.set_string(def.key, row.get_text())`. Call reset/modified helpers. NOTE: For EntryRow, add_suffix works for the reset button -- the apply button is internal.

7. `create_auto_row`: Use `scanner.get_key_info(def)` to get `SettingsSchemaKey`. Check `get_value_type().dup_string()`: "b" -> dispatch to switch, "s" -> check `get_range()` for "enum" type and dispatch to combo (extract values from range Variant) or entry, "i"/"u" -> dispatch to spin_int (check range for min/max), "d" -> dispatch to spin_double. If type unknown, return null. See research Pattern 7 (AUTO Widget Hint Resolution) for the exact range parsing approach.

**Reset and modified state (private static helpers):**

`add_reset_button(Adw.ActionRow row, SettingDef def, GLib.Settings settings)`:
- Create `Gtk.Button.from_icon_name("edit-undo-symbolic")` with `valign = CENTER`, `add_css_class("flat")`, `tooltip_text = "Reset to default"`
- Set `visible = (settings.get_user_value(def.key) != null)` -- only show when modified
- On clicked: call `settings.reset(def.key)` -- that's all, the changed signal handler below updates visibility
- `row.add_suffix(btn)`

`track_modified_state(Adw.ActionRow row, SettingDef def, GLib.Settings settings, Gtk.Button reset_btn)`:
- Set initial CSS class: if `settings.get_user_value(def.key) != null`, call `row.add_css_class("setting-modified")`
- Connect `settings.changed[def.key]` signal: check `get_user_value()`, toggle `setting-modified` CSS class and `reset_btn.visible` accordingly
- For SWITCH rows using bind(): after reset, the bind auto-updates the widget. For COMBO/SPIN rows with manual notify handlers: the changed signal also fires the notify which updates the widget value. No manual widget refresh needed -- GSettings changed signal fires which updates CSS, and for bind() rows the widget auto-syncs.

**Guard against signal loops (Pitfall 2):**
For COMBO and SPIN rows, the `notify["selected"]`/`notify["value"]` handlers write to settings, which fires `changed`, which updates CSS. This is fine -- there's no loop because the CSS update doesn't trigger another write. The risk is only if a widget value change handler reads from settings and writes back -- avoid that pattern. The one edge case: after `settings.reset()`, the changed signal fires, which could trigger the notify handler on COMBO/SPIN if the widget value updates. For SWITCH rows, `SettingsBindFlags.DEFAULT` handles this. For COMBO rows, set `combo.selected` in the changed handler only if the new value differs from current selection to avoid re-entrancy. For SPIN rows, same approach with `row.value`.

**Combine add_reset_button and track_modified_state into one helper** to keep the code clean -- call it `attach_reset_and_tracking(row, def, settings)` which creates the button, adds it as suffix, sets initial state, and connects the changed signal.
