<task type="auto">
  <name>Task 1: Create SettingDef data model and supporting types</name>
  <files>src/core/setting-def.vala</files>
  <action>
Create `src/core/` directory and `src/core/setting-def.vala` inside the `ShadowSettings` namespace.

Define the `WidgetHint` enum with values: AUTO, SWITCH, COMBO, SPIN_INT, SPIN_DOUBLE, FONT, ENTRY, CUSTOM.

Define the `SettingDef` struct with these fields:
- `string schema_id` -- GSettings schema ID (e.g., "org.gnome.desktop.interface")
- `string key` -- GSettings key name (e.g., "font-name")
- `string label` -- Human-readable title for the UI row
- `string? subtitle` -- Optional description (null = use schema summary)
- `string category` -- Category name for sidebar grouping (e.g., "Desktop", "Appearance", "Windows", "Input", "Power", "Privacy")
- `string group` -- Sub-group within category for PreferencesGroup title (e.g., "Shell", "Top Bar", "Fonts")
- `string category_icon` -- Icon name for sidebar (e.g., "preferences-desktop-wallpaper-symbolic")
- `WidgetHint widget_hint` -- Which widget to use (AUTO = infer from schema metadata)
- `double spin_min` -- Min value for SPIN_INT/SPIN_DOUBLE (default 0)
- `double spin_max` -- Max value for SPIN_INT/SPIN_DOUBLE (default 100)
- `double spin_step` -- Step increment for spin rows (default 1)
- `int spin_digits` -- Decimal places for spin rows (default 0)
- `string[]? combo_labels` -- Human-readable labels for COMBO (null = use schema enum values)
- `string[]? combo_values` -- Values corresponding to combo_labels
- `double display_factor` -- Multiplier for display transform (e.g., 1.0/60.0 for seconds-to-minutes), default 1.0

Define the `CategoryInfo` struct with:
- `string id` -- Category identifier (lowercase, e.g., "desktop")
- `string title` -- Display title (e.g., "Desktop")
- `string icon` -- Icon name
- `SettingDef[] settings` -- Array of settings in this category

Define the category ordering constant:
```vala
public const string[] CATEGORY_ORDER = {
    "desktop", "appearance", "windows", "input", "power", "privacy"
};
```

IMPORTANT: If Vala compiler rejects `const` arrays of structs with nullable fields, use `static` instead of `const`. The research flagged this as Pitfall 1 -- nullable types in const struct arrays may not compile.

All types go in the `ShadowSettings` namespace (no sub-namespace needed).

<task type="auto">
  <name>Task 2: Create SchemaScanner and CategoryMapper, update meson.build</name>
  <files>src/core/schema-scanner.vala, src/core/category-mapper.vala, meson.build</files>
  <action>
**SchemaScanner (src/core/schema-scanner.vala):**

Create a class in ShadowSettings namespace with:
- Private field: `GLib.SettingsSchemaSource source` initialized from `GLib.SettingsSchemaSource.get_default()` in constructor
- `public SettingDef[] scan (SettingDef[] registry)` -- Iterates registry, checks each entry with `source.lookup(def.schema_id, true)` then `schema.has_key(def.key)`. Returns only entries where both checks pass. This is the core FR-1 runtime detection.
- `public GLib.SettingsSchemaKey? get_key_info (SettingDef def)` -- Returns the SettingsSchemaKey for a given SettingDef, or null if schema/key doesn't exist. MUST call `has_key()` before `get_key()` (Pitfall 2 from research: get_key() aborts on nonexistent key).
- `public int total_scanned` property (set during scan) -- tracks how many registry entries were checked
- `public int total_available` property (set during scan) -- tracks how many passed validation

**CategoryMapper (src/core/category-mapper.vala):**

Create a class in ShadowSettings namespace with:
- `public CategoryInfo[] map (SettingDef[] available)` -- Groups the available SettingDef array by category field. Uses a HashTable<string, GenericArray<SettingDef?>> to collect settings per category. Then iterates CATEGORY_ORDER to build the output array in the correct sidebar order. Categories with zero settings are omitted entirely from the output.
- Each CategoryInfo in the output gets its `id` from the category name lowercased, its `title` from the first SettingDef's category field in that group, and its `icon` from the first SettingDef's category_icon field.
